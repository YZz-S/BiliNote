# 413 请求体过大问题排查与修复说明

## 背景

在 BiliNote 生成笔记流程中，任务会依次经历以下阶段：

1. 下载音频或视频
2. 语音转写
3. LLM 总结生成 Markdown
4. 截图/链接后处理
5. 保存结果

本次问题出现在第 3 步，即 `SUMMARIZING` 阶段。典型日志如下：

```text
GPT 总结失败：<html>
<head><title>413 Request Entity Too Large</title></head>
...
<hr><center>openresty</center>
```

后续在加入自动分块重试后，又出现新的运行时异常：

```text
TypeError: TranscriptSegment.__init__() missing 1 required positional argument: 'end'
```

本文档用于汇总问题现象、根因判断、修复方案、当前限制以及后续建议。

## 问题现象

### 现象 1：摘要阶段失败，状态写入 `FAILED`

任务前序步骤可以正常完成：

- 视频网格图生成正常
- 音频下载正常
- Whisper 转写正常
- 状态从 `TRANSCRIBING` 进入 `SUMMARIZING`

但在调用兼容 OpenAI SDK 的摘要接口时，请求被拒绝，导致任务失败。

### 现象 2：日志中出现 413 错误页

错误体不是 JSON，而是 HTML：

```html
<html>
<head><title>413 Request Entity Too Large</title></head>
<body>
<center><h1>413 Request Entity Too Large</h1></center>
<hr><center>openresty</center>
</body>
</html>
```

这说明拦截发生在 HTTP 网关或反向代理层，而不是模型返回的正常业务错误。

### 现象 3：分块重试后出现 `TranscriptSegment` 构造失败

在第一次修复中，系统已经能识别 413 并自动进入分块重试：

```text
检测到 413 请求体过大，开始自动分块降载重试
分块重试仍触发 413，继续降低 chunk_limit=60000
...
```

但在合并分块摘要结果时，代码使用：

```python
TranscriptSegment(start=0, text=summary)
```

而 `TranscriptSegment` 的数据结构要求同时提供：

- `start`
- `end`
- `text`

因此触发新的本地代码异常，导致任务依然失败。

## 根因分析

## 一、413 是哪一层报的

`413 Request Entity Too Large` 不是 LLM 在“理解完输入以后”返回的业务错误，而是请求尚未正常到达模型推理阶段前，由链路中的网关层直接拒绝。

从错误页中的 `openresty` 可以判断，至少当前链路中存在基于 Nginx/OpenResty 的请求体大小限制。

也就是说，这里受到的是“HTTP 请求体大小”限制，而不是单纯的“模型上下文窗口”限制。

## 二、为什么大上下文模型也可能报 413

不同模型确实拥有不同上下文容量，例如：

- 1M 上下文
- 256k 上下文
- 128k 上下文

但是“模型能接受多少 token”与“网关允许提交多大的 HTTP body”是两件不同的事：

### 模型上下文限制

- 决定模型理论上最多能处理多少 token
- 是模型服务本身的能力边界

### HTTP 请求体限制

- 决定你的请求能不能先通过代理层
- 常受 Nginx/OpenResty/API Gateway/中转服务限制

因此，即使模型支持超大上下文，只要请求在进入模型前就被网关拦下，仍然会收到 413。

## 三、旧实现为什么不够

旧版 `backend/app/gpt/universal_gpt.py` 中虽然已经有一个固定阈值：

```python
MAX_INPUT_LENGTH = 120000
```

但它存在几个问题：

1. 这是固定值，没有针对不同模型自动调整
2. 它更接近“字符预算”，不是严格的 token 预算
3. 它无法感知实际链路中的网关 body 限制
4. 即使分块，也可能仍然超过代理层可接受范围

因此单纯依赖固定阈值不足以覆盖不同模型和不同中转链路。

## 解决思路

本次采用的是“两段式修复”。

## 第一段：遇到 413 时自动降载分块重试

当首次摘要请求触发 413 时，不直接失败，而是进入自动重试流程：

1. 识别 413 或 `Request Entity Too Large`
2. 按当前阈值切分转写分段
3. 对每个分块单独发起摘要请求
4. 如果仍然 413，则继续降低分块阈值
5. 最后对分块摘要结果进行二次汇总

这样做的目标是：

- 尽量避免整条任务直接失败
- 兼容不同代理层的 body 大小限制
- 不要求用户手工裁剪输入

## 第二段：让分块阈值支持自动适配和手动配置

由于不同模型、不同 API 中转链路容忍度不同，本次把“最大输入字符预算”改为：

### 自动适配

按模型名进行保守估算，例如：

- `deepseek-v4`、`deepseek-r1`、`deepseek-chat` 默认给更高预算
- `gpt-4.1`、`gpt-5`、`claude`、`gemini` 给中高预算
- `qwen`、`gpt-4o`、`gpt-4-turbo` 使用默认预算

注意，这里的预算本质上是“prompt 文本字符预算”，不是模型官方 token 上限。

它的目标不是完全等于模型上下文，而是选择一个更稳妥、更不容易触发网关 413 的工程值。

### 手动配置

支持通过环境变量手动覆盖：

```env
BILINOTE_GPT_MAX_INPUT_LENGTH=80000
BILINOTE_GPT_MIN_CHUNK_LENGTH=4000
```

这样可以根据不同部署环境快速调整：

- 如果中转层限制较严，可以把最大长度调低
- 如果希望分块更细，可以降低最小分块长度
- 如果网关已放宽，可以适当调大

## 实际代码改动

本次主要修改文件为：

- `backend/app/gpt/universal_gpt.py`

关键改动如下。

## 1. 增加 413 识别函数

用于判断异常是否属于“请求体过大”：

- `_is_request_too_large_error()`

## 2. 增加按长度切分转写段落的逻辑

用于根据当前预算把 `TranscriptSegment` 列表切分为多个块：

- `_split_segments_by_length()`

## 3. 增加分块摘要与递归汇总逻辑

用于对多个块分别生成摘要，再将局部摘要进一步整合：

- `_summarize_with_chunk_limit()`

## 4. 修复分块汇总时的结构体构造错误

由原来的：

```python
TranscriptSegment(start=0, text=summary)
```

修正为：

```python
TranscriptSegment(start=0, end=0, text=summary)
```

从而满足 `TranscriptSegment` 的完整字段要求。

## 5. 引入自动/手动输入预算配置

新增：

- `_resolve_max_input_length()`
- `_resolve_min_chunk_length()`

用于：

- 读取环境变量
- 按模型名估算默认预算
- 给分块重试和截断逻辑提供统一入口

## 当前结论

## 1. 413 不是模型“主动拒绝总结”

它属于链路层限制，通常是：

- Nginx/OpenResty
- API Gateway
- 兼容 OpenAI 协议的中转层

在请求进入模型前就被拦截。

## 2. 大模型上下文能力不能直接等价为可提交请求大小

即使模型支持 256k、1M 上下文，也仍可能因代理层 body 限制而触发 413。

## 3. 现有方案已经从“固定阈值”升级为“自动适配 + 手动可调 + 413 分块重试”

这比原先的单一常量方案更适合多模型、多中转环境。

## 4. 新出现的 `TranscriptSegment` 报错属于本地代码缺陷，已修复

该问题与模型无关，也不是转写数据本身损坏导致。

## 验证情况

已完成以下本地验证：

1. 修改后文件静态诊断无报错
2. 执行以下命令通过：

```bash
python -m py_compile backend/app/gpt/universal_gpt.py
```

说明当前修复至少通过了语法和基础编译检查。

## 已知限制

当前方案虽然能显著降低失败率，但仍有以下限制：

### 1. 字符预算不等于真实 token 预算

当前实现基于字符串长度进行保守估算，并没有接入 tokenizer 做精确 token 统计。

### 2. 自动适配基于模型名推断

这是一种工程启发式方案，不是调用模型元数据接口得到的精确值。

### 3. 如果网关 body 限制极小，仍可能失败

即使不断缩小分块，在极端严格的代理配置下依然可能触发 413。

### 4. 多模态内容会放大请求体

如果请求中同时包含：

- 大量转写文本
- 多张 `image_url`
- 额外格式控制 prompt

那么实际 body 增长速度会更快，更容易碰到代理层限制。

## 建议的后续优化

## 方案 A：把阈值配置下沉到模型级别

当前手动配置使用环境变量，适合快速修复，但粒度不够细。

更推荐后续做成“每个模型独立配置”，例如：

- `model_name`
- `max_input_length`
- `min_chunk_length`
- `context_window`
- `body_budget_hint`

这样可以让前端或后台对不同模型分别管理策略。

## 方案 B：接入 tokenizer 做 token 级估算

相比字符长度，token 估算更接近模型真实限制。

可用于：

- 更准确判断单次请求是否超限
- 更稳定地进行分块
- 更合理利用大上下文模型能力

## 方案 C：区分“模型上下文限制”和“网关请求体限制”

后续可以把两类限制分别建模：

- 模型最大上下文
- 当前链路最大 body 预算

最终取两者中的更保守值。

## 方案 D：优化多模态摘要策略

对于带图片的摘要任务，可以进一步优化为：

1. 首轮仅发送文本分块摘要
2. 二轮只发送关键图片做补充理解
3. 最终再合并文本与视觉结论

这样可以大幅降低一次性提交的 body 大小。

## 推荐运维配置

如果当前环境经常碰到 413，建议先采用偏保守参数：

```env
BILINOTE_GPT_MAX_INPUT_LENGTH=80000
BILINOTE_GPT_MIN_CHUNK_LENGTH=4000
```

如果仍然报 413，可以继续降低：

```env
BILINOTE_GPT_MAX_INPUT_LENGTH=60000
BILINOTE_GPT_MIN_CHUNK_LENGTH=3000
```

如果部署方可以控制网关，也建议同时检查：

- `client_max_body_size`
- 代理层请求体限制
- 中转服务自身的 body 或 payload 上限

## 总结

本次问题的本质不是“模型上下文不够”，而是“摘要请求在链路层先被 body 限制拦截”，同时首次修复又暴露了一个本地结构体构造缺陷。

目前系统已经具备以下能力：

1. 识别 413 错误
2. 自动分块降载重试
3. 支持按模型名自动估算输入预算
4. 支持通过环境变量手动调整预算
5. 修复分块汇总阶段的 `TranscriptSegment` 构造错误

这套方案能明显提升长视频、长转写、多模态摘要场景下的稳定性，并为后续做模型级配置和 token 精确估算提供基础。
